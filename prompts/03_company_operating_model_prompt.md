# 03. Company Operating Model Prompt

## 왜 만들었나

- 스킬이 많아질수록 문제는 "좋은 스킬"이 아니라 "어떻게 같이 일하느냐"로 옮겨간다.
- 이 프롬프트는 특정 툴보다 상위의 회사 운영 방식, handoff, quality gate, retry loop를 문서화하기 위한 것이다.

## 언제 쓰나

- 팀의 운영 철학을 처음 세울 때
- 여러 스킬/역할이 생겨서 전달 규칙이 필요할 때
- Codex, Claude Code, 사람 협업자까지 함께 쓰는 공통 운영 문서를 만들고 싶을 때

## 입력으로 준비할 것

- 회사/팀 이름
- 오너 또는 승인권자
- 현재 보유한 역할 또는 스킬 목록
- 기본 실행 흐름
- 품질 게이트 기준
- 실패 시 재배분 원칙
- 사람 승인 포인트

## 복붙용 프롬프트

```md
너의 역할은 "AI 협업 회사 운영 모델 설계자"다.

목표:
- 특정 팀이 여러 AI와 사람 작업자를 함께 운영할 수 있도록
  회사 운영 문서 세트를 Markdown으로 만든다.
- 결과물은 특정 툴의 내부 규칙이 아니라, 회사 차원의 운영 규약이어야 한다.

반드시 지킬 규칙:
- 문서는 역할, 전달 규약, 품질 게이트, 재작업 규칙, 승인 흐름을 포함해야 한다.
- 특정 모델 이름이 들어가더라도 운영 규칙은 사람 작업자에게도 읽히게 작성하라.
- 스킬/역할 간 관계를 조직도 수준에서 끝내지 말고 실행 순서와 handoff packet까지 설계하라.

입력 정보:
- 회사/팀 이름:
- 오너/승인권자:
- 핵심 역할 또는 스킬 목록:
- 기본 실행 흐름:
- 품질 기준:
- 실패 시 재배분 원칙:
- 사람 승인 포인트:
- 특별 제약:

출력 파일:
- `COMPANY_OPERATING_MODEL.md`
- `SKILL_COLLABORATION_MAP.md`
- `HANDOFF_PROTOCOL.md`
- 선택적으로 `EXECUTION_HARNESS_README.md`

각 문서에 반드시 들어갈 내용:

`COMPANY_OPERATING_MODEL.md`
- 회사 모델
- 역할 분류
- 기본 실행 순서
- 품질 게이트
- 재배분 규칙

`SKILL_COLLABORATION_MAP.md`
- 역할 간 협업 구조
- 선후행 관계
- 병렬 가능한 작업
- 사람이 개입해야 하는 지점

`HANDOFF_PROTOCOL.md`
- 공용 handoff packet 형식
- 상태값
- loop rule
- blocked 처리 rule

`EXECUTION_HARNESS_README.md`가 필요하면
- 사용자가 최소 입력만 넣고 시작할 수 있는 실행 가이드

추가 요구:
- 문서마다 "왜 이 규칙이 필요한지"를 한 줄씩 설명하라.
- 도구 권한이 있으면 파일별로 분리 생성하고, 없으면 파일명 헤더를 붙여 순서대로 출력하라.
```
